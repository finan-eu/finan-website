# Changelog - 2025-11-24 20:08 (Europe/Helsinki)

## Added

### Screen Reader Accessibility for External Links

- **Implemented WCAG 2.1 compliant accessibility labels** for all external links opening in new tabs
- **Files Modified**:
  - `src/components/Navbar.astro` (4 locations: desktop and mobile "Guides & Resources" and "News & Events" links)
  - `src/components/Blog.astro` (2 locations: blog post titles and "View All" button)
- **Implementation**: Added `<span class="sr-only">(opens in new tab)</span>` to inform screen reader users
- **Impact**: Improves accessibility for visually impaired users, meets WCAG 2.1 Success Criterion 3.2.5

### Empty State Fallback for Blog Component

- **Added user-friendly empty state message** when Ghost CMS returns no blog posts
- **File Modified**: `src/components/Blog.astro`
- **Implementation**: Conditional rendering displays a centered message card when `posts.length === 0`
- **Message**: "No news or events are currently available. Check back soon for updates!"
- **Handles scenarios**:
  - Ghost API unavailable
  - Missing environment variables
  - No published posts
  - API errors
- **Impact**: Better UX instead of showing an empty grid with no feedback

### CSP Documentation for Image Source Wildcards

- **Added comprehensive JSDoc documentation** explaining security trade-offs for broad CSP wildcards
- **File Modified**: `security.config.js` (lines 24-63)
- **Documentation includes**:
  - Purpose explanation for each img-src wildcard pattern
  - Security warning about `*.cloudfront.net` being particularly broad
  - 4-step optimization guide for narrowing wildcards
  - Trade-off justification explaining Ghost CMS requirements
  - Mitigation strategies for future improvements
- **Updated security checklist** with two new items:
  - `⚠️  CSP img-src wildcards documented (Ghost CMS requirement)`
  - `💡 Consider narrowing CloudFront CSP after identifying specific distribution IDs`
- **Impact**: Transparent documentation of security decisions for future maintainers

## Technical Details

### Security Configuration Architecture

The security configuration now follows a centralized pattern:

```javascript
// security.config.js (single source of truth)
export const contentSecurityPolicy = { /* ... */ };
export const cspString = /* converted to string */;
export const securityHeaders = { /* ... */ };

// astro.config.mjs (imports and uses)
import { cspString, securityHeaders } from './security.config.js';

// src/layouts/Layout.astro (imports and uses)
import { cspString } from '../../security.config.js';
```

### Accessibility Pattern

The `sr-only` class pattern was already established in the codebase (e.g., Navbar.astro:185), and this update extends consistent usage to all external links.

### Blog Component Defensive Programming

The empty state handling complements the existing defensive error handling in `src/lib/ghost.ts` that returns empty arrays when Ghost API fails or isn't configured.
